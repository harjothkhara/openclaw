## Findings (ranked)

### F01-1: splitShellArgs drops empty quoted arguments and misparses `#` after an empty-quoted word as a comment

- **Priority:** P1
- **Location:** `src/utils/shell-argv.ts:18-23` (pushToken), `src/utils/shell-argv.ts:69` (comment check)
- **Description:** The tokenizer tracks "word started" implicitly via `buf.length === 0`.
  Two divergences from POSIX follow from that one root cause:
  1. A quoted empty word (`""` or `''`) produces an empty buffer, so `pushToken`
     silently discards it. POSIX shells pass an empty argv entry.
  2. A `#` immediately after a quoted empty word (`''#tail`) sees `buf.length === 0`
     and treats it as a comment, truncating the rest of the line. POSIX only starts a
     comment at the start of a word; `''#tail` is the single word `#tail`.
  The file's own header comment states the goal is keeping "command-risk analysis
  byte-faithful". Both behaviors break that: positional guards like
  `parseOpenClawChannelsLoginShellCommand` (`src/infra/exec-control-command-guard.ts:86-90`)
  match `argv[1]`/`argv[2]`, so `openclaw "" channels login` is classified as a
  channels-login command even though a real shell would pass `""` as the first
  subcommand; and the `''#` truncation hides trailing argv content from
  `buildCommandPayloadCandidates` entirely.
- **Repro / failing-test sketch:** (run on `8d9ce35b92`, Node 24)
  - `splitShellArgs('git commit -m ""')` → `["git","commit","-m"]`; bash/shlex → `['git','commit','-m','']`
  - `splitShellArgs('openclaw "" channels login')` → `["openclaw","channels","login"]`; real argv has `""` at index 1
  - `splitShellArgs("echo ''#tail more args")` → `["echo"]`; bash argv → `['echo','#tail','more','args']`
- **Skeptic verification:** CONFIRMED. Fresh pass tried to refute:
  - *"Callers depend on dropping empties"* — walked all six import sites
    (`exec-control-command-guard.ts`, `exec-approvals-analysis.ts`, `command-carriers.ts`,
    `command-analysis/risks.ts`, `bash-tools.exec.ts`, `invoke-system-run-plan.ts`); all
    consume argv positionally or scan tokens; none filters or expects empty-token
    elision. `risks.ts:72` re-joins argv with spaces, which was already lossy for
    quoting, so preserved empties don't regress it.
  - *"Existing tests pin this"* — `utils-misc.test.ts:57-78` covers quotes, escapes,
    unterminated input, and comments; no test touches empty quoted words. The pinned
    cases (`hi#tail` literal, `# comment` break) remain satisfied by a word-start-aware fix.
  - *"Some shells behave this way"* — no; verified against bash 3.2 and Python `shlex`.
  - **Upstream dupe-check:** `gh search prs/issues` on openclaw/openclaw for
    `splitShellArgs`, `shell-argv`, "empty quoted argument shell": no open or merged
    fix. Open PR #84172 (Tree-sitter command planner) reworks the broader
    authorization area but does not fix this tokenizer. Last commits touching the
    file are docs-only.
- **Disposition:** promoted to `factory/fix-f01-1`.

### F01-2: stripInlineDirectiveTagsForDisplay leaves doubled spaces where a tag is removed

- **Priority:** P2
- **Location:** `src/utils/directive-tags.ts:85-95`
- **Description:** `parseInlineDirectives` replaces tags via
  `replacementPreservesWordBoundary`, but the display-strip path replaces with `""`,
  so `"a [[audio_as_voice]] b"` renders as `"a  b"` (double space).
- **Repro / failing-test sketch:** `stripInlineDirectiveTagsForDisplay("a [[audio_as_voice]] b").text === "a  b"`.
- **Skeptic verification:** NOT RUN (not promoted this cycle). Callers may normalize
  whitespace downstream; needs a caller walk before promotion.
- **Disposition:** backlog.
